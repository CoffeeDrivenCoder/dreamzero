## 1. 本轮结论

本轮已经完成了执行侧的关键对齐：客户端使用 `direct-48`、`30Hz`、Savitzky-Golay 平滑，动作 chunk 的真实流式执行时间稳定在 `1.60s`，完整 chunk 执行时间稳定在约 `1.80s`。说明此前“动作展开过慢”的问题已经基本解决。

当前主要瓶颈已经转移到模型推理侧。除首步 warmup 外，稳态推理中位数约 `6.36s`，远高于 `1.6s` 的 action chunk 执行时长，导致整个闭环周期约 `8.2s`。机器人虽然按论文节拍执行每个 chunk，但新动作更新频率仍远低于论文实时闭环目标。

因此下一步优化目标不应优先放在执行器或平滑参数上，而应集中在：

1. 降低推理时延；
2. 验证观测输入分布；
3. 用生成视频判断失败来自视觉规划还是动作执行。

## 6. 下一步优化目标

### P0：确认 4-step 是否真实生效并记录到日志

代码已改为读取 checkpoint 中的 `num_inference_timesteps=4`，并在 4-step 下每步都跑 DiT。但当前 action JSON 没有记录 server 侧的 `num_inference_steps` 和 `DIT Compute Steps`。

下一步应把 server 侧每次推理的以下指标写入 action JSON 或独立 server metrics log：

- `num_inference_steps`
- `dit_compute_steps`
- text encoder 耗时
- image encoder 耗时
- VAE 耗时
- diffusion 耗时
- scheduler/其它耗时

验收标准：

- 每次请求能看到 `num_inference_steps=4`
- 每次请求能看到 `DIT Compute Steps=4`
- 能区分 VAE、DiT、scheduler 的耗时占比

### P0：降低稳态推理时延

当前稳态推理中位数约 `6.36s`，仍远高于 `1.6s` chunk 执行时间。

建议阶段目标：

| 阶段 | 目标 |
|---|---|
| 短期 | 稳态推理压到 `3s` 内 |
| 中期 | 让推理接近或低于 `1.6s` |
| 长期 | 支持真正异步闭环，chunk 边界不等待推理 |

推荐实验顺序：

1. 当前配置继续作为 baseline：`4帧 @ 5FPS`
2. 测试低延迟输入：`1帧 @ 30FPS`
3. 测试更贴近代码特殊路径的输入：`9帧 @ 30FPS`

每组至少记录 10 个稳态 chunk，比较：

- 生成视频是否更合理；
- 稳态推理是否下降；
- 真机是否更容易完成擦拭；
- 动作是否更抖或更不连续。

### P1：建立生成视频与 action log 的一一对应

现在视频文件和 action JSON 需要靠时间戳人工对应。建议后续增加：

- 在 action JSON 中追加 `video_rollout_path`
- 或在 mp4 文件名中加入 `session_id` 前 8 位和 prompt 简写

这样排查失败案例会更快。

### P1：暂不优先开启异步

当前 `speculative_enabled=False`。即使开启，由于推理约 `6.36s` 而执行约 `1.8s`，预取大概率无法在 chunk 边界前完成，会退化为等待或同步。

建议先优化推理时延，再开启：

```bash
--enable-speculative-inference
--speculative-prefetch-fraction 0.0
```

异步真正有收益的前提是：推理时延接近或低于 chunk 执行时间。

### P1：针对擦拭任务做任务级诊断

擦拭类任务比 pick-and-place 更依赖：

- 接触高度；
- 持续接触；
- 横向轨迹；
- 餐巾纸是否被夹稳；
- 桌面污渍是否被视觉模型正确 grounding。

建议用生成视频先判断模型是否计划了“接触桌面并横向擦拭”。如果计划正确但真机失败，再优先检查：

1. 餐巾纸抓取是否牢固；
2. 右臂下压高度是否足够；
3. Savitzky-Golay 是否过度削弱接触动作；
4. 当前动作限幅是否限制了擦拭幅度；
5. gripper close/open 阈值是否适合餐巾纸。
