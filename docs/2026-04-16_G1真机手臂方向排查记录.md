## 已确认事实

1. 真机输入三路图像在保存结果上看是正常的，目标物体在头部和双腕相机中均可见。
2. 当前重点只关注双臂问题，暂不关注夹爪、头部、腰部。
3. `one_step_can_actions.json` 中保存的是模型输出动作序列，不包含推理当时机器人当前 `arm_joint_states()` 原始值。
4. 当前客户端会把模型返回的整段动作序列全部 replay 执行；这会放大错误趋势，但更像放大器，不足以解释“第一反应方向就错”。
5. 最终 checkpoint 的 `experiment_cfg/conf.yaml` 中，AgiBot 的 `eval_delta_indices` 是 `[0]`，因此当前单帧输入与最终导出配置并不冲突。

## 当前分析结论

### 1. `metadata.json` 的含义

checkpoint 目录中的 `experiment_cfg/metadata.json` 是训练时从数据集 metadata 自动汇总出的统计结果，不是手写规则，也不是额外人工定义。

它主要包含：

- `statistics.state`
- `statistics.action`
- `modalities`

用途主要是：

- 输入 state/action 归一化
- 输出 action 反归一化

因此它可以视为“训练分布快照”。

### 2. 为什么左右臂统计分布不完全对称

虽然从机器人几何直觉上，左右臂某些关节可能近似互为负号，但训练分布不一定呈现严格镜像。可能原因包括：

- 左右臂 joint space 本来就不是逐维简单负号关系
- 数据集中左右手使用频率和任务角色不平衡
- 左右臂在不同视角、遮挡、自碰撞约束下形成了不同策略

因此 `metadata.json` 中左右臂分布不完全对称，本身不说明数据有问题。

### 3. 目前最可疑的方向

当前最值得怀疑的是：

- 左右臂语义在某处被交叉解释
- 或 live arm state 所在坐标基与训练分布不一致

这个怀疑来自 `one_step_can_actions.json` 第一行动作与训练分布的对比：

- 预测出的左臂目标的符号模式，更像训练集里的右臂分布
- 预测出的右臂目标的符号模式，更像训练集里的左臂分布

这不构成最终定论，但已经是较强异常信号。

## 已降级的方向

以下方向目前不再认为是根因，只作为放大器或次要项：

1. 一次执行整段 horizon
2. 单帧输入
3. prompt 太弱
4. 夹爪/头部/腰部数据异常

说明：

- 执行整段 horizon 会放大错误，但不能解释第一反应趋势本身错误。
- 单帧输入可能削弱后续纠偏，但不能直接解释第一步就远离目标。
- 实际 prompt 为“用左手抓取绿色易拉罐”的英文，优先级降低。

## 待验证项

当前最关键、最有信息量的待验证项只有两项：

1. 推理当时客户端原始 `arm_joint_states()` 的 14 维数值
2. 服务端该步收到的：
   - `observation/left_arm_joint_position`
   - `observation/right_arm_joint_position`

验证目标：

- 判断客户端原始 state 是否已经偏离训练分布
- 判断服务端接收到的左右臂 state 是否与客户端一致
- 判断问题出在输入映射，还是出在模型对视觉/手臂语义的联合解释

## 当前建议

后续优先做以下两类验证：

1. 记录一次真机推理时的原始 `arm_joint_states()` 14 维数值。
2. 对照服务端日志中的左右臂 state，检查是否存在左右语义错位。

在拿到上述数据前，不建议继续围绕执行步数、单帧、多帧、夹爪等方向发散。
