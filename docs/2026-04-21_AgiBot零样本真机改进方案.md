# AgiBot Zero-Shot 真机逐步改进方案

日期：2026-04-21

这份文档用于整理当前 `DreamZero-AgiBot` 真机 zero-shot 部署，与论文目标之间的主要差距，以及接下来更稳妥的逐步改进顺序。

当前判断基于以下材料：

- `DreamZero.pdf`
- `robot_live_client.py`
- `socket_test_optimized_AR.py`
- `groot/vla/model/n1_5/sim_policy.py`
- `groot/vla/model/dreamzero/action_head/wan_flow_matching_action_tf.py`
- `checkpoints/DreamZero-AgiBot/experiment_cfg/conf.yaml`

## 一句话结论

当前系统已经具备 DreamZero-AgiBot 的主要推理骨架：

- `48` 步 action chunk
- `4-step` 推理
- `3` 视角输入
- `5 FPS + 4 帧 history` 在线 observation
- causal 推理路径
- client 已新增 `direct-48` 执行模式，可按 `30 Hz` 直接下发模型原始 `48` 行动作
- client 已新增 arm-only `Savitzky-Golay` action smoothing，初步真机反馈为正收益

最新进展是：此前最大的执行时序问题已经有了可用修复路径。旧的 Ruckig 路径会把 `48` 行动作展开成上千个点，导致单个 chunk 执行十几到二十多秒；现在 `direct-48 + 30 Hz` 已经实现并完成初步真机尝试，chunk 执行节奏可以接近论文假设的 `1.6s`。

但距离论文里的真机效果，仍然差在几个关键闭环细节上：

- 推理和执行还不是真正异步
- 抓取类任务中接触点、靠近路径、闭爪时机仍不稳定
- `Savitzky-Golay` 平滑已实现第一版，但还需要系统 benchmark 不同参数
- 当前推理本身仍在约数秒量级，整体闭环还不是论文式异步并行

因此，下一步不建议一口气大改，而是按“先建立稳定基线，再逐项缩小论文差距”的方式推进。

- `action_horizon = 48`

- 多视角 observation 已接入 AgiBot server wrapper

- client 已实现 `direct-48` 模式：
  - `--arm-execution-mode direct-48`
  - `--direct-control-hz 15/30`
  - `30 Hz` 下 `48` 行动作约 `1.6s` 执行完
- client 已实现 arm-only action smoothing：
  - `--action-smoothing savgol`
  - `--savgol-window 7`
  - `--savgol-polyorder 2`
  - `--savgol-upsample 2`

- 模型内部已有 causal KV cache / `current_start_frame` 机制
- server/client 已经形成可运行的真机闭环

这说明当前问题不是“完全没复现 DreamZero”，而是“已经复现骨架，但关键闭环工程细节还没有完全对齐论文”。

## 改进原则

后续改进建议遵循下面三个原则：

1. 先改最影响精度和时序一致性的部分。
2. 先做能稳定量化收益的改动，再做复杂并行化改动。
3. 每一步都保留日志和对比基线，避免修改后只凭主观感觉判断好坏。

## Step 0：先建立稳定基线

目标：

- 让后续每次改动都能横向对比
- 避免“这次感觉好一点”但无法确认到底是哪项改动起作用

建议做法：

- 固定一套 server 启动命令
- 固定一套 client 启动命令
- 固定几个代表性 prompt
- 固定相机视角、物体位置、机器人起始姿态
- 每次测试至少记录 5 到 10 个 chunk 的日志

建议记录指标：

- 单 chunk 推理时延
- 从开始推理到动作开始执行的时延
- 一个 chunk 的真实执行时长
- 是否成功接近目标
- 是否成功闭爪
- 是否出现明显远离、抖动、停滞

验收标准：

- 后续任何改动都能与同一套 baseline 结果直接比较

## Step 1：先验证 `DiT cache` 是否真的生效

优先级：P0

原因：

`--enable-dit-cache` 理论上应该直接影响推理时延，但目前从代码静态检查上，还没有看到非常直接的完整闭环证据，暂时不能把它当成“已确认有效”。

建议做法：

- 保持其余参数完全不变
- 只切换 `--enable-dit-cache`
- 连续跑 5 到 10 个 chunk
- 对比 steady-state 的平均推理时延和方差

观察重点：

- 首轮推理是否慢很多
- 稳态 chunk 是否明显加速
- 开关 cache 后动作是否明显变差

验收标准：

- 可以明确回答 `enable-dit-cache` 是否真实带来可重复的时延收益

## Step 2：把双臂执行改成更贴近模型原始 `48` 步

优先级：P0

原因：

当前 client 不只是“执行模型输出”，而是对双臂目标又做了一层 Ruckig 分段插值和重定时。这会改变真实执行时序，使其偏离训练分布。

当前风险：

- 模型输出的 `48` 行动作，不再和机器人实际执行节拍一一对应
- 真实执行时长可能比理论 `1.6s` 长很多
- 最后靠近目标的动作趋势会被“拉平”或“拖慢”

建议做法：

- 保留安全限幅和异常值裁剪
- 减少额外重插值
- 让真实执行更接近“模型 48 行动作逐行下发”
- 先只改双臂，不同时重构腰部和底盘

验收标准：

- 日志中能明显看出“模型 48 行动作”和“机器人实际发送动作”更接近一一对应
- 单个 chunk 执行时长更接近理论时长

当前状态：

- 已完成第一版 `direct-48` 执行模式。
- 启动参数：

```bash
--arm-execution-mode direct-48
--direct-control-hz 30
```

- 该模式每个模型 row 下发一次 `move_arm`，不再做 Ruckig 分段重插值。
- 当前应保留旧的 `ruckig` 模式作为对照，不建议删除。

## Step 3：引入统一执行 scheduler，向 `30 Hz` 靠近

优先级：P0

原因：

论文假设的是：

- `48` 步 action chunk
- `30 Hz` 控制频率
- `1.6s per chunk`

当前系统虽然能执行手臂、夹爪、腰部、轮子，但这些模块并没有严格共享同一个控制时钟。

建议做法：

- 建立统一 tick，例如每 `33ms` 一个执行周期
- 每个 tick 统一决定手臂、夹爪、腰部、轮子的当前 row
- 尽量避免某些部件提前发送完、另一些部件明显滞后

验收标准：

- 执行层存在统一固定拍点
- 各执行部件共享同一个 chunk row index
- chunk 总时长稳定接近理论值

当前状态：

- 已通过 `direct-48` 模式实现第一版统一 row tick：
  - arm、gripper、waist、wheel 共享同一个 row index
  - `direct-control-hz=30` 时目标为 `48 / 30 = 1.6s`
- 仍需后续确认：
  - SDK 实际是否能稳定跟上 `30 Hz`
  - 快速执行时是否带来额外抖动
  - 是否需要在 `20/24/30 Hz` 之间选择更稳的折中点

## Step 4：实现论文里的 chunk 级动作平滑

优先级：P1

原因：

当前主要依赖：

- delta clipping
- 限幅
- Ruckig

这些不等价于论文第 9 页明确提到的 `Savitzky-Golay` 平滑。

建议做法：

- 先对 `14` 个手臂关节做平滑
- 再对腰部 `2` 维尝试平滑
- 先不要一开始就对轮子和平头部也做

建议顺序：

- `2x` 上采样
- Savitzky-Golay filter
- 下采样回原始 `48` 步

验收标准：

- 高频抖动明显减少
- 整体动作趋势不被破坏
- 抓取末端靠近时更平稳

当前状态：

- 已完成第一版 arm-only `Savitzky-Golay` smoothing。
- 启动参数：

```bash
--action-smoothing savgol
--savgol-window 7
--savgol-polyorder 2
--savgol-upsample 2
```

- 只平滑双臂 `14` 维关节：
  - 不平滑 gripper，避免改变闭爪时机
  - 不平滑 waist/head/wheel，先减少影响面
- 日志中已记录：
  - `raw_actions`
  - `actions`
  - `action_smoothing_duration_seconds`
  - `savgol_window / savgol_polyorder / savgol_upsample`
- 初步真机反馈：该优化有效，应保留为当前推荐 baseline。

## Step 5：补“逐行执行一致性”日志

优先级：P1

原因：

当前已经能看到：

- 模型输出了什么
- 机器人大致动了什么

但还缺少最关键的一层：

- 模型第 `i` 行动作
- client 第 `i` 行实际发给 SDK 的动作
- 机器人执行后第 `i` 行附近的真实回读状态

只有把这层做清楚，才能真正判断：

- 是模型输出就不对
- 还是 client 改坏了动作
- 还是机器人没按预期跟上

建议日志项：

- row index
- 该 row 的左右臂动作
- 该 row 的夹爪命令
- 该 row 的腰部命令
- 下发后的短时间回读状态

验收标准：

- 可以逐行比较“模型输出”和“机器人真实执行”的一致性

当前状态：

- 已有 chunk 级时延、首末行、post-state 日志。
- 还没有逐行记录每个 row 的模型输出、实际下发动作和短时回读状态。
- 由于当前主要问题已经转向抓取质量，建议把该项提前到下一批优化中。

建议新增一版轻量 row log：

- 默认关闭，通过 `--log-row-execution` 打开。
- 每个 row 记录：
  - `row_index`
  - safe left/right arm target
  - gripper command
  - wheel command
  - waist command
  - row send timestamp
- 不建议默认每行都读回机器人状态，先避免拖慢 `30 Hz` 执行；可以每 `N` 行或 chunk 末尾读回。

## Step 6：在同步版本稳定后，再做真正异步执行

优先级：P2

原因：

论文最核心的闭环优势来自：

- 当前 chunk 执行时
- 下一 chunk 同时推理

但这一步工程复杂度高，也最容易把调试难度抬高。

因此建议：

- 先把同步版本做稳
- 先保证 `48` 步动作落地可信
- 再做异步并行

建议做法：

- 维持一个执行线程
- 维持一个推理线程
- chunk 边界只切换“下一块动作指针”
- 真值 observation 在 chunk 边界更新

验收标准：

- 当前 chunk 执行时，下一 chunk 能同时推理
- 总体闭环延迟明显下降

## Step 7：最后再考虑 DreamZero-Flash 路线

优先级：P2

原因：

当前你面对的主要问题不是“4-step 一定不行”，而是：

- 执行时序没完全对齐
- 闭环还不是论文式异步
- 动作落地仍有额外工程变形

这些问题不解决，直接切 Flash 容易把系统复杂度再抬高一层。

建议做法：

- 等前面步骤都稳定后
- 再考虑 decoupled noise / 更少 DIT steps / 1-step 推理

验收标准：

- 时延明显下降
- 动作质量仍然可接受

## 当前最推荐的实际推进顺序

如果按收益、难度和风险综合排序，我建议当前按下面顺序做：

1. 固定 `direct-48 + 30 Hz` 作为当前主 baseline，同时保留 `ruckig` 作回退对照
2. 固定 arm-only `Savitzky-Golay` 作为推荐 baseline，并保留 `--action-smoothing none` 做对照
3. 验证 `DiT cache` 与推理时延稳定性，确认当前 7-9 秒级推理是否还有可压缩空间
4. 针对抓取任务补逐行执行日志，先确认闭爪时机和接触点
5. 调整夹爪策略：研究是否需要 gripper 阈值、闭合保持、或根据任务阶段延迟/提前闭爪
6. 做抓取 prompt/场景基线：纸杯、易拉罐、衣服分别固定 2-3 个短 prompt
7. 再做真正异步执行
8. 最后再考虑 Flash 路线

## 当前如果只做一件事，最推荐做什么

如果现在只允许优先做一项，我最推荐：

### 优先验证 `DiT cache` 和推理时延稳定性

原因：

- 执行时序已经通过 `direct-48 + 30 Hz` 初步对齐
- arm-only 平滑已经有正收益，控制层当前可作为 baseline
- 下一项最贴近论文复现的是确认 DiT cache 真实收益和推理时延瓶颈
- 若推理仍需 7-9 秒，整体闭环仍明显慢于论文式实时执行

简单说：

当前最值得优先修的，不再是“chunk 太慢”或“动作太硬”，而是“模型推理是否已经跑到当前硬件/配置能达到的最快稳定状态”。如果推理时延确认无法再低，下一步再进入异步执行。

## 后续讨论建议

后面继续联调时，建议每次都按下面五层来定位问题：

1. 模型看到了什么
2. 模型输出了什么
3. server 转换成了什么
4. client 实际下发了什么
5. 机器人真实执行成了什么

只要这五层还没有同时打通，就容易把不同层次的问题混在一起误判。
