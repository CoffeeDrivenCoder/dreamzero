# 文档规范

本文档规定 `docs/` 目录下所有文档的命名、结构和写作约定。新增文档前请先阅读本文件。

---

## 文件命名

格式：`YYYY-MM-DD_中文标题.md`

- **时间戳放在文件名最前面**，日期格式固定为 `YYYY-MM-DD`。
- 日期取文档**首次创建**的日期，后续更新不改文件名（在文件内记录更新日期）。
- 日期后使用一个下划线连接中文标题，标题应简短、可读、能直接说明文档用途。
- 文件名必须使用中文命名；代码名、模型名、机器人名等专有名词可保留英文，例如 `AgiBot`、`DROID`、`DreamZero`。
- 不使用空格、驼峰式英文长标题或旧式分类前缀（如 `PLAN_`、`DEBUG_`、`GUIDE_`）。

**推荐标题类型**（不再放进文件名前缀，只体现在中文标题里）：

| 类型 | 标题写法示例 |
|------|-------------|
| 操作指南、步骤文档 | `2025-11-01_DreamZero新机器人数据转换与训练指南.md` |
| 架构设计、方案设计 | `2026-04-27_AgiBot论文一致性对齐计划.md` |
| 问题排查记录 | `2026-04-27_AgiBot抓取高度估计问题诊断.md` |
| 改进计划、路线图 | `2026-04-22_VLA定向增强科研实现路线图.md` |
| 参考资料、命令速查 | `2026-04-21_AgiBot运行命令速查.md` |
| 流程/规范类 | `2026-04-27_文档规范.md` |

**示例**：

```
2025-11-01_DROID数据转换指南.md
2026-04-27_AgiBot论文一致性对齐计划.md
2026-04-13_AgiBot服务排障记录.md
2026-04-21_AgiBot运行命令速查.md
```

## 文档结构

每份文档须包含以下四个固定部分，顺序不变：

### 1. 标题与元信息

文件第一行是一级标题，紧跟元信息块：

```markdown
# 文档标题（简洁描述主题）

**日期**：YYYY-MM-DD  
**状态**：`草稿` / `有效` / `已过期`  
**关联文件**：相关代码文件或其他文档的路径（可选）
```

**状态**字段含义：
- `草稿`：内容未确认，不可直接依赖。
- `有效`：当前准确，可作为参考依据。
- `已过期`：内容已不再适用，保留仅供历史溯源。

### 2. 一句话结论

每份文档在元信息块之后必须有一个**一句话结论段**，不超过 3 行，让读者不读全文就能知道这份文档的核心结论或最重要的操作。

```markdown
## 结论

（一句话或至多三行，说明这份文档的核心结论/操作要点。）
```

### 3. 正文

正文自由组织，但需遵守以下规则：

- **用中文写作**。代码、命令、路径、变量名保留原始英文。
- 用 `##` 和 `###` 分节，不得跳级（不能从 `##` 直接跳到 `####`）。
- 代码块须标注语言（`` ```bash ``、`` ```python ``、`` ```yaml `` 等）。
- 路径和文件名用反引号包裹：`` `socket_test_optimized_AR.py` ``。
- 不要写"TODO"或"待补充"之类的空占位，有内容再写。
- 不要写未经验证的推测，若有不确定内容，明确标注"**待验证**"。

### 4. 更新日志（可选，超过一次修改时必须有）

```markdown
## 更新日志

| 日期 | 修改内容 |
|------|---------|
| YYYY-MM-DD | 初始版本 |
| YYYY-MM-DD | 补充 XXX 部分 |
```

## 文档类型细则

### 操作指南

- 步骤必须可直接复制执行，不能有"根据实际情况修改"之类的模糊指示。
- 每个步骤说明**为什么**，不只是**做什么**。
- 末尾必须有"验收标准"小节，说明如何判断操作是否成功。

### 排查记录

- 必须包含：问题描述、已确认事实、根因分析、已处理内容、待验证项。
- "已确认事实"和"推测"严格区分，推测须标注"**待验证**"。
- 问题解决后将状态改为`已过期`，并在更新日志里注明。

### 改进计划

- 每项改动须说明：做什么、为什么、预期效果、实现难度。
- 分优先级排列，优先级须有依据，不能只是主观排序。
- 计划落地后，对应条目标注"**已完成 YYYY-MM-DD**"，不删除原内容。

### 命令速查

- 所有命令须经过实际验证，不得写"参考命令"。
- 命令中的可变参数须用 `<PLACEHOLDER>` 标注并附说明。
- 注明命令的适用环境（服务器/客户端、哪个 conda 环境）。

---

## 语言与风格

- **写结论，不写过程**。"分析了 X 之后发现 Y"改写为"Y（因为 X）"。
- **不写冗余的背景铺垫**。读者已了解项目背景，直接进入主题。
- **数字具体化**。"推理较慢"改为"稳态推理约 6.6 s"。
- **不用"目前"、"当前"之类的时间副词描述代码状态**，这些描述会随时间失效；改为"截至 YYYY-MM-DD"或直接引用文件行号。

---

## 过期处理

文档内容失效时：

1. 将文件头的`状态`字段改为`已过期`。
2. 在文件头补充一行：`**被取代文档**：<新文档路径>`。
3. **不删除文件**，保留历史溯源。
