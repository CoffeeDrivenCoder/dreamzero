# VLA 定向增强科研实现路线图

## 0. 目标与边界

这项工作的核心目标不是单纯复现 `VLA-RL`，而是实现一个 **failure-driven targeted augmentation / curriculum framework**：

- 基座策略：`OpenVLA-OFT`
- RL 后训练底座：优先 `SimpleVLA-RL`
- benchmark：`LIBERO` 和 `RoboTwin 2.0`
- 主贡献：根据策略失败模式，自动决定下一批应强化的场景因子，而不是做静态或随机 domain randomization

第一版工作的边界需要刻意收紧：

- 先不追求一个统一 student 同时 joint train `LIBERO + RoboTwin 2.0`
- 先不把 `world model / WAM` 放进主训练闭环
- 先不把真机在线 RL 作为核心贡献
- 先把方法做成：**统一 teacher 接口 + benchmark-specific adapter + simulator-backed RL training**

一句话概括第一阶段论文故事：

> 我们提出一个 failure-aware 的场景定向增强框架，让 VLA 在 RL 后训练中自动补齐最脆弱的分布外场景。

## 1. 冻结的第一版技术选择

为了保证可实现性，第一版先冻结以下决策：

1. RL 训练框架优先用 `SimpleVLA-RL`
2. backbone 先固定为 `OpenVLA-OFT`
3. `LIBERO` 与 `RoboTwin 2.0` 先分别训练、分别评测，只共享 teacher 方法与因子空间
4. teacher 先不用 RL teacher，先用 `bandit / adaptive sampler`
5. reward 不做复杂重设计，先沿用 benchmark 原有成功信号或简洁任务奖励
6. world model 作为第二阶段扩展，不进入第一版主线

如果后续需要切到 `RLinf`，应该发生在：

- `SimpleVLA-RL` 已经能稳定跑通
- targeted augmentation 已经在至少一个 benchmark 上有效
- 我们明确需要更统一的 simulator / algorithm / world-model backend

## 2. 总体阶段划分

整个项目建议分成 8 个阶段推进，每个阶段都必须有明确交付物和退出条件。

### Phase 0. 基础设施与复现实验准备

目标：

- 把后续研究需要的代码、权重、benchmark、日志结构先搭平

必须完成的事：

1. 确认 `SimpleVLA-RL` 在本地环境可安装、可运行
2. 下载或整理 `OpenVLA-OFT` 的 SFT 权重
3. 跑通 `LIBERO` 的最小训练/评测脚本
4. 跑通 `RoboTwin 2.0` 的最小训练/评测脚本
5. 确定实验目录规范：
   - `configs/`
   - `scripts/`
   - `logs/`
   - `analysis/`
   - `docs/research/`
6. 确定统一日志字段，至少包括：
   - benchmark
   - task
   - seed
   - factor config
   - success
   - episode length
   - reward
   - checkpoint

交付物：

- 一份环境安装记录
- 一份权重与 benchmark 依赖清单
- 两个 benchmark 的 smoke test 结果

退出条件：

- `LIBERO` 和 `RoboTwin 2.0` 都能各自完成至少一次最小评测
- 至少能从脚本中读出 success rate 和 episode 结果

失败止损点：

- 如果 `RoboTwin 2.0` 环境依赖过重，先只保留数据与评测能力，把 RL 训练主线放在 `LIBERO`

### Phase 1. 纯基线复现：LIBERO

目标：

- 先在 `LIBERO` 上拿到一个可靠的 VLA-RL 基线

建议范围：

- 不要第一天就跑完整 130 tasks
- 优先从 `LIBERO-Spatial` 或 `LIBERO-Object` 小范围开始
- 跑通后再扩到 `Goal / Long / 90`

必须完成的事：

1. 复现 `OpenVLA-OFT + SimpleVLA-RL` 的基本训练流程
2. 建立 checkpoint 管理与评测脚本
3. 记录：
   - baseline SFT 成绩
   - RL 后训练成绩
   - 不同任务组的表现差异
4. 补全 rollout 日志，至少保存：
   - 成功 / 失败
   - 任务 id
   - 指令
   - 场景随机种子

交付物：

- 一组可重复的 `LIBERO` baseline 曲线
- 一份 benchmark 行为观察记录

退出条件：

- 至少在一个 task group 上稳定提升
- 多次运行结果波动可接受

### Phase 2. 纯基线复现：RoboTwin 2.0

目标：

- 在 `RoboTwin 2.0` 上建立第二个基线，验证方法具有 benchmark 可迁移性

建议范围：

- 不要一开始挑战全部 50 个任务
- 先选 3-7 个官方已有 RL 结果或 SFT 对比明显的任务
- 优先覆盖不同失败类型：
  - 抓取类
  - 双臂协作类
  - 放置类

必须完成的事：

1. 跑通 `SimpleVLA-RL` 的 `RoboTwin 2.0` 训练脚本
2. 梳理官方已有 domain randomization 配置项
3. 确认哪些场景因子可以直接被 adapter 调控

RoboTwin 2.0 当前已知可直接利用的因子：

- clutter
- lighting
- background
- tabletop height
- language variation
- camera-related settings

交付物：

- 一组 `RoboTwin 2.0` baseline 任务结果
- 一份可控因子清单

退出条件：

- 至少 3 个任务可以稳定评测
- 已明确 targeted augmentation 可以挂载到哪些 config 字段

### Phase 3. 统一因子空间与 benchmark adapter

目标：

- 把“出题人”从 benchmark-specific trick 提炼成统一接口

第一版建议统一因子空间：

```text
z = {
  lighting,
  clutter,
  viewpoint,
  support_surface,
  language,
  object_pose,
  occlusion_level
}
```

实现原则：

- `teacher` 只输出统一因子配置
- 每个 benchmark 通过 `adapter` 翻译成具体环境参数

必须完成的事：

1. 定义 `factor schema`
2. 实现 `LIBERO adapter`
3. 实现 `RoboTwin 2.0 adapter`
4. 定义 clean / random / targeted 三种采样模式

交付物：

- 一个与 RL trainer 解耦的 factor interface
- 两个 adapter 的配置映射表

退出条件：

- 给定同一份 `z`，两个 benchmark 都能生成对应场景配置

### Phase 4. Failure bank 与 failure analyzer

目标：

- 让 teacher 不再基于直觉出题，而是基于失败模式出题

第一版 failure label 不要太复杂，建议先分三类：

- `perception failure`
- `planning failure`
- `execution failure`

每条 rollout 最少记录：

- benchmark
- task
- factor config
- success / failure
- failure stage
- episode length
- instruction
- random seed

实现原则：

- 第一版可用启发式打标签
- 先保证标签“可用”，再追求标签“精细”

交付物：

- 一个 failure bank
- 一个按 factor bucket 聚合失败率的分析脚本

退出条件：

- 可以回答下面两个问题：
  1. 这个模型最常在哪类场景失败？
  2. 哪些因子组合最影响成功率？

### Phase 5. Teacher：从静态增强到定向增强

目标：

- 实现第一版真正的方法模块

第一版 teacher 建议：

- 不做复杂 RL teacher
- 先做 `contextual bandit / UCB / Thompson sampling`

输入：

- 最近窗口内成功率
- factor bucket 失败率
- task 难度
- clean-vs-hard gap

输出：

- 下一批训练的 factor bucket / factor mixture

建议训练采样配比：

- `50%` teacher-targeted hard cases
- `30%` clean nominal cases
- `20%` random exploration

必须完成的事：

1. 实现 teacher score 函数
2. 实现 factor bucket sampler
3. 将 sampler 接到 RL rollout 之前
4. 验证 targeted 训练不会明显伤害 clean success

交付物：

- targeted augmentation 训练闭环

退出条件：

- 至少在一个 benchmark 上优于：
  - random domain randomization
  - static hard-case oversampling
  - vanilla RL finetuning

### Phase 6. 两个 benchmark 上的方法验证与消融

目标：

- 把方法从“能跑”变成“能证明”

必须做的对照：

1. `SFT only`
2. `Vanilla VLA-RL`
3. `Random DR`
4. `Static hard-case replay`
5. `Ours: failure-driven targeted augmentation`

必须做的消融：

1. 不同 factor 子集
2. 不同 teacher 更新频率
3. 不同 clean/hard mixing ratio
4. 不同 failure label 粒度
5. 单 benchmark 训练与跨 benchmark 共享 teacher 规则

核心指标：

- clean success
- OOD success
- robustness gain
- data efficiency
- forgetting on nominal distribution

交付物：

- 主结果表
- 消融表
- 至少 3 张关键图：
  - failure profile before/after
  - teacher sampling distribution evolution
  - clean vs OOD tradeoff

退出条件：

- 结果能支持论文主张，而不是只在个别任务上有效

### Phase 7. 真机验证

目标：

- 用最少真机成本验证 simulator 中发现的脆弱因子确实对应现实脆弱性

原则：

- 真机是验证平台，不是第一版主要训练平台
- 只选 2-4 个最能体现 targeted augmentation 价值的任务

推荐验证方式：

1. 用 baseline policy 跑真机，收集失败场景
2. 将真机失败映射回 factor space
3. 在 simulator 中定向补课
4. 再回到真机验证是否改善

不要一开始就做的事：

- 大规模真机在线 RL
- 真机上频繁重训 student

交付物：

- 真机前后对比视频与表格
- 真机失败 -> simulator 定向增强 -> 真机恢复 的 case study

### Phase 8. World model / WAM 扩展支线

目标：

- 把 world model 作为 teacher 加速器，而不是第一版主训练环境

推荐定位：

- `teacher-side candidate screening`
- `difficulty estimation`
- `short imagined rollout ranking`

不推荐第一版的做法：

- 直接让 student 在 imagined world 中主训练

原因：

- 容易引入 hallucination
- 很难归因性能提升来自 targeted augmentation 还是 imagined simulator 偏差
- 工程和论文逻辑都会显著复杂化

只有在下面条件满足时才进入这条线：

1. simulator-backed targeted augmentation 已经成立
2. world model 能稳定做短时预测
3. 我们明确需要降低 teacher 搜索成本

## 3. 推荐时间线

这是一个偏稳健、适合科研推进的时间安排。

### 第 1 周

- 环境与依赖核验
- `SimpleVLA-RL` 跑通最小例子
- `LIBERO` smoke test
- `RoboTwin 2.0` smoke test
- 整理 experiment logging 规范

### 第 2-3 周

- 完成 `LIBERO` baseline 复现
- 建立统一评测脚本
- 初步观察失败模式

### 第 4-5 周

- 完成 `RoboTwin 2.0` baseline 复现
- 梳理 domain randomization 可控接口
- 确定统一 factor schema

### 第 6-7 周

- 实现 adapter
- 实现 failure bank
- 实现第一版 teacher sampler

### 第 8-9 周

- 先在 `LIBERO` 上跑 targeted augmentation
- 调 teacher mixing ratio
- 做第一轮消融

### 第 10-11 周

- 在 `RoboTwin 2.0` 上验证方法可迁移性
- 形成 benchmark 间统一结论

### 第 12 周及以后

- 真机验证
- world model teacher-side 加速器
- 论文写作与图表整理

## 4. 论文视角下的最小可投稿单元

如果中间资源或时间紧张，最小可投稿单元建议是：

- benchmark：`LIBERO + RoboTwin 2.0`
- 架构：`OpenVLA-OFT + SimpleVLA-RL`
- 方法：failure-driven targeted augmentation
- 验证：
  - 至少一个 benchmark 上完整主结果
  - 另一个 benchmark 上迁移验证
  - 一个小规模真机 case study

如果必须进一步收缩，优先保留：

1. `LIBERO` 主实验
2. `RoboTwin 2.0` 小规模扩展验证
3. world model 暂不纳入

## 6. 第一阶段验收标准

第一阶段真正算“开工成功”，需要满足以下条件：

1. `SimpleVLA-RL + OpenVLA-OFT + LIBERO` 能稳定训练或评测
2. `RoboTwin 2.0` 至少能稳定做小规模评测
3. 已确定统一因子空间雏形
4. 已有可追踪的 rollout 日志
5. 已能看到至少一种明确失败模式

## 7. 我们现在的最优推进策略

从科研收益与工程风险平衡来看，最优顺序是：

1. 先把 `SimpleVLA-RL` 跑通
2. 先在 `LIBERO` 上建立稳定 baseline
3. 再把 `RoboTwin 2.0` 当作 robustness 验证台接上
4. 再做 failure bank 与 teacher
5. 最后再考虑 world model / WAM 是否值得作为 teacher-side 加速器

不要反过来做：

- 不要先做 world model
- 不要先追求统一 joint training
- 不要先做复杂 reward engineering

因为这些都会削弱可实现性，并拖慢真正的方法验证。
