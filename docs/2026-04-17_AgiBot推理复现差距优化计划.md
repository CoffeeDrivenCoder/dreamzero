# DreamZero AgiBot 推理复现差距与优化优先级

日期：2026-04-17

这份文档用于整理当前 `DreamZero-AgiBot` 在线推理复现，与论文目标之间的主要差距，并给出一个兼顾收益、实现难度和调试风险的执行顺序。

当前判断基于以下代码与配置：

- `robot_live_client.py`
- `socket_test_optimized_AR.py`
- `groot/vla/model/n1_5/sim_policy.py`
- `groot/vla/configs/data/dreamzero/base_48_wan_fine_aug_relative.yaml`
- `checkpoints/DreamZero-AgiBot/experiment_cfg/conf.yaml`

## 一句话结论

当前这套系统更像是“能跑起来的工程闭环版本”，还不是“严格按论文时序和推理约束对齐的复现版本”。

和论文差距最大的部分，不是模型权重本身，而是：

- 动作语义链路是否和我们理解一致
- 视觉时序输入是否严格按 `5 FPS` 和历史帧组织
- 推理与执行是否异步并行
- 执行时序是否接近 `30 Hz / 1.6s per chunk`
- 是否使用论文中的动作平滑与实时优化手段

## 已经对上的部分

- `action_horizon = 48`
- `num_action_per_block = 48`
- `num_inference_timesteps = 4`
- 使用 Wan2.1 系列 VAE / text encoder / image encoder
- 使用 causal 推理路径
- 使用 session_id 区分并在 session 切换时 reset
- 多视角输入包含 `top_head / hand_left / hand_right`

这些说明当前系统已经具备 DreamZero AgiBot 的基础推理骨架。

## 仍然没有严格对上的部分

- 当前主循环仍然是“推理完再执行”的串行模式，不是论文里的异步并行
- 当前控制执行不等于严格的 `30 Hz`
- 当前没有实现论文里提到的 `Savitzky-Golay` 动作平滑
- server 会先把相对动作还原成绝对目标，再发给 client 执行
- `DiT cache` 虽已纳入推荐启动方式，但还没有完成收益验证
- 当前不是 DreamZero-Flash 路线，而是标准 DreamZero 的 `4-step` 推理
- 虽然 `4` 帧历史 observation 已进入 server，但模型前输入 shape 还没有做最终强验证日志

## 优先级列表

下面按 `优先级 / 难度 / 收益 / 风险` 综合排序。

### P0-1 动作语义链路打透

优先级：P0

难度：低

收益：极高

原因：

当前最大的不确定性之一，是“模型输出到底是我们以为的 delta，还是已经在 server 里被还原成 absolute target”。如果这一点没有彻底看清，后面很多现象都会被误判，比如：

- 为什么机械臂会先远离目标
- 为什么看起来像某个关节正负号错了
- 为什么模型输出看似合理，但真机表现不合理

当前代码里，`sim_policy.unapply()` 会把 relative action 加回 last_state，得到 absolute target。

建议动作：

- 在 server 和 client 两侧补充完整日志
- 同时记录 `normalized action`
- 同时记录 `unnormalized relative action`
- 同时记录 `unapply 之后的 absolute action`
- 同时记录 `最终发给机器人 SDK 的动作`

验收标准：

- 可以对任意一步动作明确回答“模型原始输出是什么、server 变换后是什么、robot 实际收到的是什么”

### P0-2 严格改成 5 FPS 在线视觉输入

优先级：P0

难度：低

收益：高

原因：

论文对 `5 FPS` 输入是硬约束。当前 client 更像是“循环里取最新图像”，并没有严格控制送给模型的视觉节奏。

这会带来两个问题：

- 时序分布可能和训练时不一致
- 模型看到的相邻帧变化速度可能不稳定

建议动作：

- 在 client 中加入固定 `5 FPS` 节流
- 明确每 `200ms` 采样一次图像
- 不再使用“谁最新就拿谁”的策略作为实际模型输入

验收标准：

- 日志中可看到连续 observation 的图像时间间隔稳定在约 `0.2s`

当前状态：

- 已完成 client 侧固定 `5 FPS` 采样
- 目前更推荐把剩余工作放在“验证采样是否稳定”和“确认多帧输入对行为的实际收益”

### P0-3 显式构造历史帧输入，而不是只送当前帧

优先级：P0

难度：中

收益：高

原因：

当前 AgiBot wrapper 没有像 Droid wrapper 那样显式维护多帧历史 buffer。虽然模型内部可能有 causal cache，但在 client/server 这一层，没有明确把“最近几帧真实观测”显式组织成输入。

这件事很可能直接影响策略是否能正确理解“目标越来越近还是越来越远”。

建议动作：

- 在 AgiBot client 或 server wrapper 里加入固定长度图像 buffer
- 先从最近 `4` 帧真实图像开始
- 保证这些帧来自真实观测，而不是预测视频

验收标准：

- 每次推理输入都能在日志中确认包含最近 `4` 帧图像
- 模型输入的时间序列来源可追踪

当前状态：

- 已完成到 server 端的真实链路验证
- 最新一批 `server_received_agibot_inputs` 已确认三路相机均为 `4` 帧输入
- 若要继续收口，可补模型前 shape 日志做最终确认

### P0-4 先把 chunk 执行时长固定下来

优先级：P0

难度：中

收益：高

原因：

当前系统是：

- 先做一次 inference
- 再执行整个 chunk
- 再等待 arm close
- 然后才进入下一轮 inference

这会把论文里的 `1.6s chunk` 执行，拉成实际更长的时间窗口。之前日志里已经出现过 `5~7s` 的推理周期，这和论文的实时闭环差距很大。

建议动作：

- 降低过长的阻塞等待
- 用固定 chunk 时间预算来调度执行
- 即使暂时不做并行，也先让每个 chunk 的执行长度更接近论文时序

验收标准：

- 单个 chunk 的完整“接收动作到完成执行”时长明显下降
- 时长稳定，不再高度依赖 `wait_close`

### P1-1 开启并验证 DiT cache

优先级：P1

难度：低

收益：中到高

原因：

这项是论文实时性优化的重要组成部分，当前代码里已经有开关，但默认关闭。

建议动作：

- 打开 `enable_dit_cache`
- 记录开启前后的单步推理耗时
- 确认缓存不会破坏当前动作稳定性

验收标准：

- 推理延迟明显下降
- 输出动作质量没有明显恶化

### P1-2 统一执行 scheduler，向 30 Hz 靠近

优先级：P1

难度：中

收益：高

原因：

当前手臂、腰部、底盘的控制节奏并不统一。系统更像“按各自函数调用时机执行”，而不是严格共享一个控制拍点。

建议动作：

- 引入统一执行 scheduler
- 每个周期按统一节拍更新手臂、腰部、头部、夹爪、底盘
- 目标先靠近 `30 Hz`

验收标准：

- 执行层存在统一的固定拍点
- 同一时刻发出的控制信号来源一致

### P1-3 增加动作平滑

优先级：P1

难度：中

收益：中

原因：

当前主要依赖：

- delta clipping
- Ruckig 插值
- 局部限幅

这和论文里明确提到的 `Savitzky-Golay` 后处理并不相同。

建议动作：

- 在 chunk 输出层增加可开关的平滑模块
- 先只对手臂和腰部做
- 不建议一开始就对底盘直接加重平滑

验收标准：

- 高频抖动降低
- 不改变整体动作趋势

### P1-4 明确 observation 和 execution 两条链路的尺度对齐

优先级：P1

难度：中

收益：高

原因：

我们已经遇到过夹爪 observation 尺度不一致的问题。类似问题理论上也可能存在于：

- 头部
- 腰部
- 末端执行器
- 底盘速度

建议动作：

- 对照 metadata 和训练配置，逐项检查 observation 尺度
- 对照 action 分布，逐项检查 execution 尺度
- 把“训练分布单位”和“真机 SDK 单位”写成表格

验收标准：

- 每个维度都能明确回答“输入单位是什么，输出单位是什么”

### P2-1 推理和执行真正并行

优先级：P2

难度：高

收益：极高

原因：

这是论文实时闭环里最关键的一项，但工程复杂度也最大。它会牵涉：

- 推理线程
- 执行线程
- 中断和急停
- session 状态同步
- chunk 边界的真值观测更新

建议动作：

- 先在前面 P0、P1 项稳定后再做
- 先做“单 producer + 单 executor”的简单版本

验收标准：

- 当前 chunk 执行时，下一 chunk 推理可同时进行
- 总体延迟明显下降

### P2-2 严格复刻论文里的 KV cache 更新策略

优先级：P2

难度：高

收益：中到高

原因：

当前系统已经是 causal 的，但“每执行完一个 chunk，就用真实观测替换 cache 中预测帧，并只保留真实观测”这一点，是否和论文 1:1 一致，还没有被严格证明。

建议动作：

- 梳理当前 cache 生命周期
- 明确哪些内容来自真实 observation
- 明确哪些内容来自模型预测
- 在 chunk 边界做严格替换

验收标准：

- 可以明确证明 cache 中保留的时序来源

### P2-3 如果目标是 Flash，再切到 1-step

优先级：P2

难度：高

收益：主要体现在速度

原因：

当前 checkpoint 路线是标准 DreamZero `4-step`，不是 Flash `1-step`。如果现在主要目标还是“让行为正确、抓取稳定”，那这项不是第一优先级。

建议动作：

- 等前面时序和动作链路对齐后再考虑
- 把 `1-step` 当成性能优化阶段，而不是当前行为问题的优先修复项

验收标准：

- 单步推理时延明显下降
- 行为质量仍可接受

## 推荐实施顺序

### 第一批：最小代价换最大收益

- `P0-1` 动作语义链路打透
- `P1-1` 开启并验证 `DiT cache`
- 验证 `P0-2 / P0-3` 对动作质量的实际收益

其中：

- `P0-2` 和 `P0-3` 的工程接入已经完成
- 第一批当前应转为“验证收益 + 补齐动作语义链路”

这批做完后，通常就能明显提升“模型理解当前场景变化”的一致性，也更容易判断问题到底在模型、输入还是执行层。

### 第二批：把执行时序拉回论文要求附近

- `P0-4` 固定 chunk 执行时长
- `P1-2` 引入统一控制 scheduler，向 `30 Hz` 靠近
- `P1-3` 增加动作平滑
- `P1-4` 检查 observation / execution 尺度对齐

这批主要解决“推理正确但执行出来不对”的问题。

### 第三批：追求真正的论文级实时闭环

- `P2-1` 推理和执行真正并行
- `P2-2` 严格复刻 KV cache 更新策略
- `P2-3` 如果需要，再切 Flash `1-step`

这批是最后冲实时性和论文一致性时再做。

## 当前最推荐的三个下一步

如果现在只允许做三件事，我建议优先做：

1. 动作语义链路完整打日志
2. 把 chunk 执行时长稳定下来，减少串行阻塞
3. 验证 `5 FPS + 4 帧历史输入 + DiT cache` 对精度和速度的真实收益

原因很简单：

- 这三项最容易做
- 这三项最影响当前行为
- 这三项能最大幅度减少后续调试中的“假问题”

## 下一步建议

考虑到当前真实进度，后续建议按下面顺序推进：

### 下一步 1：补动作语义链路日志

目标：

- 明确模型输出的是相对动作还是已经还原后的绝对目标
- 明确 server 到 client 的动作变换过程
- 避免后续再把“动作语义问题”和“执行精度问题”混在一起

建议补的日志：

- `normalized_action`
- `unnormalized_action`
- `unapply` 后的动作
- client 最终下发到 SDK 的动作

### 下一步 2：验证 `5 FPS + 4 帧 history` 的实际收益

目标：

- 判断这次时序输入改造，到底对抓取精度和动作趋势改善了多少

建议对比：

- 改前单帧输入的任务表现
- 改后 4 帧输入的任务表现
- 是否更少出现“接近目标后停止靠近”
- 是否更少出现抓取前抖动

### 下一步 3：进入 `P0-4`，缩短 chunk 阻塞时间

目标：

- 让系统更接近论文要求的实时闭环
- 避免 observation 更新虽然已经变好，但整体控制仍被串行执行拖慢

优先做法：

- 继续缩短 `arm_close_timeout`
- 控制 chunk 执行时间预算
- 减少“执行完成后才允许下一轮推理”的额外等待

### 下一步 4：做 `P1-1` benchmark

目标：

- 量化 `--enable-dit-cache` 是否真的降低了时延
- 确认它不会带来明显的动作退化

建议记录：

- 首轮推理时延
- 稳态推理时延
- 开关 `DiT cache` 前后的均值和方差

### 下一步 5：准备进入 `P1-4`

目标：

- 对 observation / action 的单位、范围、分布做系统化检查
- 尤其重点关注：
  - 夹爪
  - 腰部
  - 头部
  - 底盘速度

## 后续讨论建议

后面继续推进时，建议始终按下面顺序问问题：

1. 模型看到了什么
2. 模型输出了什么
3. server 转换成了什么
4. client 实际下发了什么
5. 机器人真实执行成了什么

只要这五层没有同时打通，很多现象都可能只是中间某一层的错位。
