## 结论

当前推理实现与论文存在 4 个关键不一致：推理步数机制（16-step scheduler + mask 而非真正的 4-step）、观测时序（4帧@5fps 而非单帧@30fps）、`num_frame_per_block` 配置值（代码默认 1 vs checkpoint 训练值 2）、异步并行触发时机（70% 而非 chunk 起点）。建议按优先级分两组修改：第一组立即对齐训练分布（推理步数、观测时序、配置值、执行节拍），第二组完善论文提到的工程细节（平滑、异步触发）。

## 当前状态分析

### 训练配置关键参数（来自 checkpoint）

从 `/data1/wangk/checkpoints/DreamZero-AgiBot/experiment_cfg/conf.yaml` 确认的训练配置：

| 参数 | 训练值 | 当前推理值 | 一致性 |
|------|--------|-----------|--------|
| `fps.agibot` | 30 | 5 (client) | ❌ 不一致 |
| `video.eval_delta_indices` | `[0]` | 4 帧历史 | ❌ 不一致 |
| `num_frame_per_block` | 2 | 1 (代码默认) | ❌ 待验证 |
| `num_action_per_block` | 48 | 48 | ✅ 一致 |
| `action_horizon` | 48 | 48 | ✅ 一致 |
| `num_inference_timesteps` | 4 | 未生效 | ❌ 不一致 |
| `decouple_inference_noise` | false | false | ✅ 一致 |
| `relative_action` | true | true | ✅ 一致 |

**关键发现**：

1. **训练时 `eval_delta_indices=[0]` 意味着推理只用当前帧（1帧），不是 4 帧历史**。
2. **训练 fps=30，每个 action 步长 33.3ms；client 当前 5fps，帧间隔 200ms，时序分布不一致**。
3. **`num_inference_timesteps=4` 在代码里没有真正控制推理循环，实际是 `num_inference_steps=16`（硬编码）+ `NUM_DIT_STEPS` mask**。

### 推理时延现状

从日志分析（`robot_json/agibot_actions_*.json`）：

- **4月16日（单帧输入）**：稳态推理约 5.3s
- **4月27日（4帧历史）**：稳态推理约 6.6s
- **差异来源**：VAE encode 4 帧比 1 帧多约 1.3s

当前 2 卡并行已是代码上限（`ip_size` 硬编码只支持 1 或 2），无法通过增加 GPU 数量降低时延。

## 第一组：立即对齐训练分布（P0）

这组改动直接影响模型看到的输入分布，必须优先修复。

### 1. 修改推理步数机制为真正的 4-step

**当前问题**：

`wan_flow_matching_action_tf.py:175` 硬编码 `self.num_inference_steps = 16`，`num_inference_timesteps=4`（config 里的值）没有被用来控制推理循环。实际推理是 16-step scheduler + `NUM_DIT_STEPS=8` 的 mask 跳步。

**修改方案**：

```python
# groot/vla/model/dreamzero/action_head/wan_flow_matching_action_tf.py:175
# 改前：
self.num_inference_steps = 16

# 改后：
self.num_inference_steps = config.num_inference_timesteps  # 使用 config 里的 4
```

同时在 `should_run_model` 方法（第 909 行）增加"严格论文模式"分支：

```python
def should_run_model(self, index, current_timestep, prev_predictions):
    # 论文模式：4-step 时每步都跑，不走 mask 跳步
    if self.num_inference_steps == 4:
        return True
    
    # 原有的 dynamic_cache_schedule 和 dit_step_mask 逻辑
    if not self.dynamic_cache_schedule:
        return self.dit_step_mask[index]
    # ...
```

**验收标准**：

- Server 启动日志显示 `num_inference_steps=4`
- 单次推理的 DiT forward 调用次数为 4（不是 8 或 16）
- 推理时延预期降低到 3-4s（从当前 6.6s）

**实现难度**：低（改 2 处代码）

**预期收益**：
- 推理时延降低约 40%（从 6.6s → 4s）
- 与论文 4-step 推理严格一致

---

### 2. 观测时序改回训练分布

**当前问题**：

Client 发送 `4帧@5fps`（帧间隔 200ms），但训练配置是 `eval_delta_indices=[0]` + `fps=30`（单帧当前观测，训练时相邻帧间隔 33ms）。

**修改方案**：

Client 启动参数改为：

```bash
--observation-history 1      # 只发当前帧
--observation-fps 30         # 采样频率改为 30fps（与训练一致）
```

**注意事项**：

- 这会让模型失去历史帧信息，可能影响动作连贯性。
- 但这是**训练分布**，必须先对齐，再评估是否需要 fine-tune 支持多帧。
- 如果单帧效果明显变差，可以考虑折中方案：`--observation-history 2 --observation-fps 15`（2帧@15fps，帧间隔 67ms，接近训练时的 `num_frame_per_block=2` 时长）。

**验收标准**：

- Server 收到的 `videos.shape[2]` 为 1（单帧）
- 推理时延降低约 1.3s（从 6.6s → 5.3s，回到 4月16日水平）
- 与修改 1 叠加后，总时延预期约 2.5-3s

**实现难度**：零（只改启动参数）

**预期收益**：
- 输入分布与训练严格一致
- 推理时延降低约 20%

---

**验收标准**：

## 第二组：完善论文工程细节（P1）

这组改动是论文提到但当前"部分实现"的点，不影响核心推理分布，可在第一组完成后再做。

### 5. Savitzky-Golay 平滑做成"严格模式"

**当前状态**：

已实现 arm-only 平滑（`--action-smoothing savgol`），参数可配置。

**建议改进**：

增加 `--action-smoothing paper_mode`，固定参数为论文推荐值：

```python
if smoothing_config.mode == "paper_mode":
    window = 7
    polyorder = 2
    upsample = 2
    # 只平滑 arm joints，不改变 gripper/waist/wheel
```

**验收标准**：

- 启动参数 `--action-smoothing paper_mode` 生效
- 日志记录平滑前后动作用于对比

**实现难度**：低

---

### 6. 异步并行触发时机改成 chunk 起点

**当前问题**：

`prefetch_fraction=0.7`，投机预取在 chunk 执行 70% 时触发，更像"晚提交"。论文风格应是 chunk 开始就推下一块。

**修改方案**：

```bash
--speculative-prefetch-fraction 0.0   # chunk 起点立即触发
```

**现实约束**：

当前稳态推理 6.6s（修改 1+2 后预期 2.5-3s），chunk 执行 1.6s。即使改成起点触发，推理仍比执行慢，`max_boundary_wait_s=0.15` 必然超时，投机预取会退化成同步。

**真正有效的前提**：

推理时延必须 < 1.6s。可能的路径：
- 修改 1+2 后时延降到 2.5-3s，仍不够。
- 需要 TensorRT 加速（工程量大）或 Flash checkpoint（1-step 推理）。

**建议**：

先完成第一组修改，确认推理时延能降到多少，再决定是否开启异步。

**实现难度**：零（只改启动参数），但收益取决于推理时延

## 不建议做的事

1. **不要尝试 4 卡扩展**：代码硬编码只支持 1/2 卡，改动成本高且收益有限。
2. **不要在对齐前就做异步并行**：当前推理时延远超 chunk 执行时间，异步无实际收益。
3. **不要同时改多个变量**：每次只改一组，确认效果后再改下一组，避免无法定位问题。
